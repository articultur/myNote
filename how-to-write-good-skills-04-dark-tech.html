横跨多个类别的 Skill 容易让人困惑——每个 Skill 应只属于一种类型

Skill 是一个文件夹系统

SKILL.md 只是入口，真正的价值在：

  • references/
    详细参考文档，按需加载
  • assets/
    模板文件，供复制使用
  • scripts/
    可执行脚本，封装复用逻辑
  • hooks/
    按需钩子，skill 激活时生效
    • Progressive Disclosure

    分层加载，只在需要时才读取引用文件

    SKILL.md 的标准格式

    每个 skill 都以 SKILL.md 作为入口文件

    SKILL.md 结构
    ---
name: skill-name
description: 触发条件
version: 1.0.0
category: 技能类别
---

    1Tool Wrapper

    让 Agent 立刻成为某个库或框架的专家

    与其把 API 约定硬编码进 system prompt，不如把这些知识封装成 skill。只有当 Agent 真正处理相关技术时，才加载上下文。
    用户 Prompt→关键词匹配→加载 references/→执行规范
    最容易实现的一种模式。把团队内部编码规范、框架最佳实践，直接分发到开发者工作流中。

    2Generator

    基于可复用模板，生成结构化文档

    Agent 每次生成的文档结构都不一样。通过「模板填空」流程，把输出稳定下来。
    加载 references/style-guide.md加载 assets/template.md
    ↓
    向用户补齐缺失变量
    ↓
    按模板填充内容并输出
    适合：API 文档、commit message、项目骨架

    3Reviewer

    按照检查清单，对代码进行评审

    传统方式
    在 system prompt 里塞进一长串规则 → 难以维护
    Reviewer 方式
    检查标准外置 → 换一套 checklist 立刻变成另一种评审
    代码→加载 checklist→逐项检查→按严重程度输出
    Python 风格检查表 → OWASP 安全检查表 = 安全审计专家

    4Inversion

    在动手之前，先由 Agent 来「采访」你

    Agent 不再立刻执行，而是先收集上下文。按阶段提问，在用户完整回答前禁止开始构建。
    阶段 1
    问题发现
    Q1 - Q3
    阶段 2
    技术约束
    Q4 - Q6
    阶段 3
    方案整合
    —
    门禁式指令：在所有阶段完成之前，不要开始构建

    5Pipeline

    通过检查点，强制执行严格的多步骤流程

    复杂任务最怕步骤被跳过、指令被忽视。Pipeline 用显式检查点确保无法绕过。
    Step 1→Step 2→Step 3→Step 4
    菱形门禁条件：上一步未确认 → 禁止进入下一步

    如何选择正确的模式？

    你的 skill 主要解决什么问题？

    你的问题对应模式
    「AI 不知道用什么 API？」→ Tool Wrapper
    「每次输出格式不一样？」→ Generator
    「检查标准不好切换？」→ Reviewer
    「AI 还没搞懂就开干？」→ Inversion
    「步骤总被跳过？」→ Pipeline
    这些模式可以自由组合：Pipeline skill 末尾加 Reviewer 步骤，或 Generator 最开始套用 Inversion 先提问收集变量。
    CHAPTER II

    进阶技巧

    编写技巧 & 案例

    渐进式披露

    Skill 是文件夹，不只是 markdown

    references/
    函数签名、用法示例，按需加载
    assets/
    输出模板，供复制使用
    scripts/
    辅助函数库，封装复用逻辑

    告诉 Claude 有哪些文件 → 它在需要时读取

    节省上下文窗口

    考虑好初始设置

    有些 Skills 可能需要用户提供上下文来完成初始设置

    config.json
    把设置信息存在 Skill 目录下。如果配置还没设置好，智能体就会向用户询问相关信息
    AskUserQuestion
    如果希望智能体向用户展示结构化的多选题，可以让 Claude 使用此工具
    范例：站会发送到 Slack → 先问用户「要发到哪个频道？」

    记忆与数据存储

    有些 Skills 可以通过在内部存储数据来实现某种形式的记忆

    简单方式
    只追加写入的日志文件（.log）或 JSON 文件
    复杂方式
    SQLite 数据库

    范例：standup-post Skill 保留 standups.log，记录写过的每条站会汇报，下次运行读取历史→知道昨天到现在 的变化

    CLAUDE_PLUGIN_DATA — 每个插件的稳定数据存储目录，升级时不会被删除

    存储脚本与生成代码

    你能给 Claude 的最强大的工具之一就是代码

    给 Claude 提供脚本和库，让它把精力花在组合编排上——决定下一步做什么，而不是重新构造样板代码。
    数据科学 Skill 范例
    提供从事件源获取数据的函数库 + 辅助函数，Claude 即时生成脚本组合这些功能，完成「周二发生了什么？」这类高级分析
    封装复用逻辑，让 Claude 专注决策而非编码

    按需钩子

    Skills 可以包含只在当前 Skill 被调用时才激活的钩子，在整个会话期间保持生效

    /careful
    通过 PreToolUse 匹配器拦截危险操作：rm -rf · DROP TABLE · force-push · kubectl delete
    /freeze
    阻止对特定目录之外的任何 Edit/Write 操作
    适合场景：主观判断——你不想一直运行，但有时候极其有用
    Claude 钩子类型：PreToolUse · PostToolUse · Stop · SessionStart

    快速构建方式

    触发时机：创建 skill、优化 skill、运行 evals 测试

    /skill-creator — 从零创建新 skill、迭代优化、运行 evals 测试
    /skill-development — 插件内 skill、钩子开发、目录结构验证
    构建流程：
    • 引导用户澄清目标 — skill 要做什么？何时触发？期望输出格式？
    • 系统化收集需求 — 边缘情况、输入输出格式、示例文件、成功标准
    • 生成 SKILL.md — name · description · compatibility · 正文
    • 生成测试用例 — 为客观可验证输出的 skill 创建 evals
    • 追踪优化效果 — 用 eval-viewer 持续改进 skill 质量
    CHAPTER III

    实战案例

    test-assistant — 需求联动助手

    混合模式：Pipeline + Tool Wrapper + Reviewer

    test-assistant 是什么

    核心价值：基于需求 ID，聚合多源信息，生成结构化测试分析文档

    Dima
    需求详情
    语雀
    设计文档
    Git
    提交记录
    评审
    评审记录
    所有结论必须追溯到 config.json 配置的来源，无证据标注"待查证"

    四大核心指令

    每个指令职责单一，Pipeline 模式强制顺序执行

    init
    →
    ingest
    →
    refetch
    →
    analyze
    init
    • 初始化工作区
    • 配置检查校验
    • MCP 就绪检测
    analyze
    • 双视角评估
    • 生成测试分析
    • Review 强制关卡

    Pipelineinit 流程：6 步强制顺序

    ⛔ 阻断式设计：上一步未完成，下一步不执行

    1. 检查 CLAUDE.md
    2. validate-config
    3. check-mcps
    4. rag-status
    5. 引导填配置
    6. 生成概览
    关键校验脚本：
    validate-config.shcheck-mcps.shrag-status.sh

    Tool Wrapper数据采集 + 分析生成

    ingest 流程
    多源数据拉取
    • Dima URL → 提取 openWorkItemId
    • 语雀 → 正文 + 图片解析
    • Git 提交 → 变更分析
    • 评审 → 缺陷定性
    analyze 流程
    双视角评估
    • Feature-Anchor：用户旅程
    • Change-Anchor：代码路径
    • Duality-Merge：合并去重
    • Review：强制关卡

    Reviewer校验脚本体系

    检查标准外置，换一套 checklist 立刻变成另一种评审

    validate-config.sh
    检测占位字符串 ("kb-xxxxx")
    init Step 2
    check-mcps.sh
    MCP 可用性检测
    init Step 3
    rag-status.sh
    Serena 索引状态
    init Step 4
    validate-test-analysis.sh
    测试分析文档完整性
    analyze Step 7

    ⚡Superpowers 强制触发

    特定场景自动调用对应技能（bundled skills）

    3+ 模块变更→brainstormingStep 3 前
    多线并行→dispatching-parallel-agentsStep 5 前
    完成前验证→verification-before-completionStep 7 前
    有审查流程→requesting-code-reviewStep 7 前

    test-assistant 设计原则

    1
    来源优先 — 所有结论必须追溯到 config.json 配置的来源
    2
    证据说话 — 无证据的结论标注"待查证"
    3
    步骤强检 — analyze 步骤必须按序执行，Review 是强制关卡
    4
    模式组合 — Pipeline + Tool Wrapper + Reviewer 协同
    设计模式不是孤立的，可以自由组合
